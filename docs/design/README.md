# Design

Working artifacts from the design pipeline. Each design effort lives in its own per-run folder, `docs/design/{slug}/`, where `{slug}` is a kebab-case name for the system being designed.

## What lives here

Three artifacts per run, one per pipeline stage:

- `use-cases.md`: required behavior, from the `behavioral-requirements` skill.
- `volatilities.md`: the volatilities list and proposed change boundaries, from the `volatility-decomposition` skill.
- `architecture.md`: the layered classification and named services, from the `architecture-layering` skill.

A run may stop after any stage, so a folder can hold one, two, or three of these.

## How it is produced

Invoke the `architecture-design-pipeline` skill to run all three stages over one `{slug}` folder, or invoke any stage skill on its own. Each stage reads the upstream artifact in the same folder if it exists and writes its own, so a stage works whether the previous stage ran or the input was written by hand. The `{slug}` is derived once from the system name and confirmed before the artifacts are written.

## Difference from ADRs and business rules

These are ephemeral working files, not durable records. They can be regenerated by rerunning the pipeline and pruned once a design has been promoted into code, ADRs, or business rules. ADRs in `../adr/` and business rules in `../business-rules/` are durable and are never deleted; design artifacts carry no such guarantee.

The decision to organize the pipeline this way is recorded in `../adr/adr-design-establish-architecture-design-pipeline.md`.
